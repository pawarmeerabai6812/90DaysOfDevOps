# Day 07 – Linux Troubleshooting Scenarios

## Overview

Today's practice focused on troubleshooting common Linux issues that DevOps Engineers encounter in production environments. The exercises covered service management, log analysis, CPU troubleshooting, and file permission issues using standard Linux commands.

---

## Environment

- **Operating System:** Ubuntu Linux (AWS EC2)
- **Shell:** Bash
- **User:** ubuntu

---

# Scenario 1 – Docker Service Not Starting

### Problem Statement

The **Docker** service failed to start after a server reboot.

## Step 1 – Check the Service Status

```bash
systemctl status docker
```

**Why**

Checks whether the Docker service is running, inactive, or failed and provides an initial error message.

---

## Step 2 – Review Service Logs

```bash
journalctl -u docker -n 50
```

**Why**

Displays the last 50 log entries to identify the root cause of the failure.

---

## Step 3 – Verify Boot Configuration

```bash
systemctl is-enabled docker
```

**Why**

Checks whether Docker is configured to start automatically during system boot.

---

## Step 4 – Review Detailed System Logs

```bash
journalctl -xe
```

**Why**

Provides detailed system logs that may contain additional information related to the service failure.

---

# Scenario 2 – High CPU Usage

### Problem Statement

The application server is responding slowly due to high CPU utilization.

## Step 1 – Monitor Live System Performance

```bash
top
```

**Why**

Displays live CPU and memory usage for all running processes.

> Press **q** to exit.

---

## Step 2 – Identify Top CPU-Consuming Processes

```bash
ps aux --sort=-%cpu | head -10
```

**Why**

Lists the processes consuming the highest CPU resources.

---

## Step 3 – View Process Details

```bash
ps -fp <PID>
```

**Why**

Displays detailed information about the selected process.

---

## Step 4 – Monitor CPU Statistics

```bash
vmstat 1 5
```

**Why**

Monitors CPU, memory, and system performance over five intervals.

---

# Scenario 3 – Finding Docker Service Logs

### Problem Statement

A developer requests the recent logs for the **Docker** service.

## Step 1 – Verify Docker Service Status

```bash
systemctl status docker
```

**Why**

Confirms that the Docker service is active and identifies any immediate issues.

---

## Step 2 – View Recent Logs

```bash
journalctl -u docker -n 50
```

**Why**

Displays the most recent 50 log entries generated by Docker.

---

## Step 3 – Monitor Logs in Real Time

```bash
journalctl -u docker -f
```

**Why**

Continuously displays new log entries as they are generated.

> Press **Ctrl + C** to stop monitoring.

---

## Step 4 – View Logs Since System Boot

```bash
journalctl -u docker --since today
```

**Why**

Displays Docker logs generated during the current day.

---

# Scenario 4 – File Permission Issue

### Problem Statement

A shell script located at `/home/user/backup.sh` returns the following error:

```text
Permission denied
```

---

## Step 1 – Check File Permissions

```bash
ls -l /home/user/backup.sh
```

**Why**

Verifies whether the execute (`x`) permission is missing.

---

## Step 2 – Add Execute Permission

```bash
chmod +x /home/user/backup.sh
```

**Why**

Grants execute permission to the script.

---

## Step 3 – Verify Updated Permissions

```bash
ls -l /home/user/backup.sh
```

**Why**

Confirms that execute permission has been successfully applied.

Expected output:

```text
-rwxr-xr-x
```

---

## Step 4 – Execute the Script

```bash
./backup.sh
```

**Why**

Runs the script and verifies that it executes successfully.

# Key Learnings

- Verified the status of systemd-managed services using `systemctl`.
- Investigated service failures using `journalctl`.
- Identified high CPU-consuming processes using `top` and `ps`.
- Monitored system performance using `vmstat`.
- Resolved file execution issues by updating file permissions with `chmod`.
- Followed a structured troubleshooting workflow:
  **Check → Investigate → Fix → Verify**
